## What You Are Building

An interactive HTML dashboard about a real dataset you care about, generated by Claude and opened in your browser. Charts you can hover over. Filters you can click. Numbers you can trust because you asked how they were computed.

You are not learning Excel. You are not learning SQL. You are not learning to code. You are learning to ask a data file good questions in plain English, and to get back something visually compelling you can actually share.

By the end you will have:

- A working HTML file on your computer with interactive charts
- A clear understanding of what is in your data and what it actually means
- A reusable prompt pattern you can run on any data file from now on

> **At work, this comes up when you're...**
>
> - handed a messy CSV or spreadsheet and needing three real insights by end of day
> - reviewing a report from your team and wondering whether the numbers really say what they appear to say
> - trying to explain a dataset to leadership without making a slide with three bullet points
> - curious about your own data (inbox, calendar, expenses, team output) and wanting to see patterns
> - preparing for a meeting where someone will ask "how did you get that number?"

## Why This Matters

Most people freeze when they see a data file. The freeze is not about the data. It is about the tool. Spreadsheets make you feel like you need to know formulas before you are allowed to ask a question.

You do not. You need to ask a clear question, verify the answer, and show the result in a way that actually helps people see it. Charts are the showing part. When Claude builds you an interactive HTML dashboard, you go from "here is a number I am claiming" to "here is a picture you can explore yourself." That changes the conversation entirely.

The second win is trust. A chart that you understand and can defend is more persuasive than a spreadsheet nobody else has time to open. When you can point at a bar, explain what it represents, and tell the room how you computed it, you are using data the way senior leaders actually use it.

## Find (or generate) a real dataset

> **Hint:** _Real data beats tutorial data every time. If you do not have anything on hand, Claude will generate a realistic sample for you, on a topic you actually care about._

Option 1: you have a real file. Grab a CSV, a pasted table, or an exported spreadsheet. Any of these work:

- A CSV export from a tool you use (sales, support, sign-ups, attendance, inventory)
- A table from a report your team produced
- A list of cases, tickets, customers, or events

Option 2: you do not have one. Ask Claude to generate a realistic sample. In your Claude Code window, paste:

```
Generate a realistic sample CSV dataset I can use to practice data analysis. Pick a topic that would match my actual work: [describe your role in one sentence, like "program manager for a nonprofit running three cohorts a year"]. Produce about 100 rows with 5 to 7 columns. Include a mix of numbers, categories, and dates. Add some realistic messiness (a few blank cells, one inconsistent category, one outlier) so it feels like real data.

Save the file to my working directory as sample-data.csv and tell me what each column represents.
```

Either way, you now have a real-feeling dataset to work with.

> **You're there when:** you have a CSV or data file ready, and you can name in one sentence what decision or question you want the data to inform.

## Let Claude describe the data before you ask it anything

> **Hint:** _Do not lead with your questions. Let Claude describe what is there first. That is where you catch bad columns, messy formats, and broken assumptions._

Paste this prompt, then share or paste the file:

```
Here is a data file I want to understand. Before any analysis, do three things in plain English:

1. Tell me what each column appears to represent, one sentence each.
2. Flag anything messy, inconsistent, or possibly wrong (blank cells, mixed formats, duplicate rows, columns that mix two different things).
3. Tell me the 5 most interesting questions this data could answer, from the perspective of someone trying to make a decision.

Do not run any analysis yet. Just describe what is here.
```

Read the response. If something does not match what you know about the data, correct it:

```
Column C is not a percentage, it is a raw count. Treat it that way from here on out.
```

This step saves hours later. A wrong assumption at the top poisons every chart downstream.

> **You're there when:** you can explain every column in one sentence and you know what is messy.

## Ask Claude to build an interactive HTML dashboard

> **Hint:** _This is the creative move. Instead of asking for a text summary, ask for a visual dashboard Claude writes as a single HTML file you open in your browser._

Paste this prompt:

```
Build me a single, self-contained HTML file that visualizes this data as an interactive dashboard. Requirements:

- Use Chart.js via CDN so the file just works when I open it in a browser
- Produce 3 to 5 charts that answer the most useful questions about this data
- Include at least one interactive filter (a dropdown, a set of buttons, or date range) so I can slice the data
- Above each chart, write a one-sentence plain-English caption that explains what it shows and how it was computed
- Use clean, modern styling (good typography, plenty of whitespace, soft color palette, no neon)
- Include a small "Data notes" section at the bottom explaining any row exclusions, cleaning steps, or caveats

Save the file as dashboard.html in my working directory. Then tell me which chart surprised you most and why.
```

Claude will write the HTML file and save it. Open the file in your browser.

You should see a real dashboard. Charts. Colors. Interactive filters. Click through the filters. Hover over the bars and points. Notice what you see.

> **You're there when:** you have `dashboard.html` open in your browser and the charts respond to your clicks and hovers.

## Interrogate every chart before you trust it

> **Hint:** _A chart that looks good is not the same as a chart that is true. Verify before you share._

Go chart by chart. For each one, paste this prompt:

```
For the [name of the chart], tell me in plain English:

1. Exactly which rows were included and which were excluded.
2. The formula or method used to compute the values (in words, not code).
3. One fast sanity check I can run in under a minute to confirm the chart is plausible.
4. Anything about how the chart is rendered (color choice, axis scale, ordering) that could be misleading on a quick glance.
```

Run the sanity check. Count a top bar by hand. Check one week against your memory. If something is off, push back:

```
The bar chart for "vendor performance" included rows with blank dates. Regenerate the chart without those rows and show me how the ranking changes.
```

```
The line chart's y-axis starts at 80 instead of zero, which exaggerates the drop. Rerender with a zero baseline so the magnitude is honest.
```

```
The pie chart is sorted randomly. Sort slices largest to smallest so the biggest categories read first.
```

Each round makes the dashboard more honest.

> **You're there when:** every chart has a caption you believe, a method you could defend in a meeting, and a visual treatment that does not mislead.

## Add one more chart that answers a question you actually have

> **Hint:** _The charts Claude suggests are a starting point. Your own real question is almost always more interesting than the defaults._

Look at your dashboard. What do you actually want to know that is not there yet?

Ask Claude:

```
Add one more chart to the dashboard. I want to see [describe the question in plain English, like "which three cohorts had the biggest retention gap in the first 30 days" or "how often do my meetings run over, broken out by day of week"].

Pick the chart type that shows this best. Add the same plain-English caption and computation note. Save and regenerate the file.
```

Refresh the HTML in your browser. You should see the new chart, exactly where you asked for it.

This is the move that turns a generated dashboard into your dashboard. Every question you add compounds your understanding of the data.

> **You're there when:** you have at least one chart in the dashboard that answers a question you brought to the data, not just one Claude suggested.

## Save the pattern as a reusable skill

> **Hint:** _You just did a specific sequence that works on almost any dataset. Bottle it once and every future data question gets 20 minutes shorter._

Paste this:

```
Turn what we just did into a reusable Claude Code skill called "data-dashboard". The skill should:

- Take any data file I paste or share
- Describe the columns, flag messiness, and propose the 5 most interesting questions
- Wait for me to confirm the questions worth answering
- Build a single HTML file with Chart.js via CDN, interactive filters, and 3 to 5 charts
- Write plain-English captions above every chart
- Include a Data Notes section at the bottom with row exclusions and caveats
- Include a "how it was computed" explanation for every number so I can defend it

Ask me 2 clarifying questions about my default style (light or dark theme, dense or spacious) and then install the skill.
```

Claude will draft the skill and install it. The next time a CSV lands in your inbox, you invoke the skill and get a shareable dashboard in 10 minutes instead of an hour of staring at columns.

> **You're done when:** you have `dashboard.html` saved, at least one chart that answers your own question, and a reusable skill installed that you could run on a new dataset tomorrow.

## What You Just Did

You just moved data work from "spreadsheet anxiety" to "interactive dashboard you can defend." Specifically, you:

1. **Described the data** before analyzing it, so bad assumptions never made it into charts.
2. **Generated a real dashboard** as a single HTML file, with interactive filters and hoverable charts.
3. **Interrogated every chart** until you could defend how each number was computed and how the visual was rendered.
4. **Added your own question** to the dashboard, turning generated output into something you actually built.
5. **Captured the pattern as a skill** so the next dataset is a 10-minute job.

This is the difference between someone who uses data and someone who makes data make sense. The dashboard is a real artifact. Share it. Embed it. Use it in your next meeting.

## What Not to Do

1. **Do not skip the describe-the-data step.** A wrong column assumption poisons every chart downstream.
2. **Do not trust a chart because it looks good.** Verify the method behind every number. Always.
3. **Do not let Claude pick a misleading visual treatment.** Zero baselines. Consistent scales. Sorted categories. If a chart leans a specific way on a glance, check whether the data actually leans that way.
4. **Do not ship the default dashboard.** Add at least one chart that answers a question you brought. That is what makes it yours.
5. **Do not stop at one dataset.** The skill only earns its keep when you use it on two or three different data files and tune the defaults to your taste.
